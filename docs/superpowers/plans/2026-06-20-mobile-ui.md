### Task 3: iOS safe-area insets

Add `env(safe-area-inset-*)` support so content isn't hidden behind the iPhone notch (top) or home-indicator (bottom). Wrapped in `@supports` so old browsers get the unchanged base `body` padding.

**Files:**
- Modify: `app/static/style.css` (append new `@supports` block near end, before the `@media (prefers-reduced-motion: reduce)` block)

**Interfaces:**
- Consumes: nothing from other tasks
- Produces: nothing consumed by other tasks

- [ ] **Step 1: Find insertion point**

```bash
grep -n 'prefers-reduced-motion: reduce' app/static/style.css
```

Note the line number — insert the new block immediately before it.

- [ ] **Step 2: Add `@supports` block**

Immediately before the `@media (prefers-reduced-motion: reduce)` block, insert:

```css
@supports (padding: env(safe-area-inset-bottom)) {
  body {
    padding-left:   max(1.25rem, env(safe-area-inset-left));
    padding-right:  max(1.25rem, env(safe-area-inset-right));
    padding-bottom: max(1.25rem, env(safe-area-inset-bottom));
  }
}
```

`max()` ensures the padding never shrinks below the base `1.25rem` even where the safe-area value is 0 (most Android phones, desktop browsers). The `@supports` guard means old browsers that don't understand `env()` skip the block entirely and keep the base `body { padding: 1.25rem }`.

Note: `padding-top` is intentionally omitted — the topbar already sits below the status bar on iOS because the `<meta name="viewport">` tag does not set `viewport-fit=cover`. If a future update adds `viewport-fit=cover` to extend into the notch, add `padding-top: max(1.25rem, env(safe-area-inset-top))` to this block at that time.

```bash

```

Expected to show the `max(1.25rem, env(safe-area-inset-bottom))` lines.
